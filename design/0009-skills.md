# AI Skill System Design

## REQ-SKILL-0001: Skill System Architecture

The AI Skill System allows extending the AI agent with specialized domain knowledge and tools without modifying the core plugin. It is modeled after the Gemini CLI skill system.

### Core Components

1.  **AISkill** (Class): Represents a single skill. It points to a directory containing:
    - `SKILL.md`: Instructions for the AI.
    - `tools/`: A directory containing GDScript files extending `AITool`.
    - `examples/`: Reference files.
2.  **SkillManager** (Utility): Handles discovery of skills in `res://ai_skills/` and `res://addons/ai_assistant/skills/`.
3.  **Skill Registry**: A way to map tool names back to the skill that provides them.

## REQ-SKILL-0002: Skill Structure

A skill folder (e.g., `res://ai_skills/godot-ui/`) should look like this:

```
godot-ui/
├── SKILL.md
├── tools/
│   ├── layout_helper_tool.gd
│   └── theme_analyzer_tool.gd
└── examples/
    └── sample_ui.tscn
```

### `SKILL.md` Injection
The contents of `SKILL.md` are injected into the system prompt when the skill is active:

```
<activated_skill name="godot-ui">
[Contents of SKILL.md]
</activated_skill>
```

## REQ-SKILL-0003: Skill Discovery & Locations

Skills are discovered by scanning two primary search paths. This separation ensures that core plugin capabilities (built-in) are preserved while allowing for project-specific extensions.

### 1. Search Paths
- **Built-in Skills**: `res://addons/ai_assistant/skills/`
    - Contains skills bundled with the plugin (e.g., `skill-creator`, `godot-docs`).
    - These are read-only from the project's perspective but essential for core functionality.
- **Project Skills**: `res://ai_skills/`
    - The default location for user-created skills and skills generated by the `skill-creator`.
    - These are project-specific and should be committed to version control.

### 2. Discovery Mechanism (`SkillManager`)
The `SkillManager` is a singleton or static utility that performs the following on startup or when requested:
1.  **Directory Crawl**: Uses `DirAccess` to list subdirectories in both search paths.
2.  **Validation**: For each folder, it verifies the existence of a `SKILL.md` file.
3.  **Indexing**: Creates a map of `skill_id` (the folder name) to the full resource path.
4.  **De-duplication**: If a skill with the same name exists in both paths, the **Project Skill** (`res://ai_skills/`) takes precedence, allowing users to "override" or customize built-in skills.

### 3. Built-in Skill: `skill-creator`
The `skill-creator` will be located at `res://addons/ai_assistant/skills/skill-creator/`. 
- By default, it will be enabled in the plugin settings to ensure the AI can always bootstrap new capabilities.
- The `skill-creator` tools will be configured to save new skills into the `res://ai_skills/` path specifically.

## REQ-SKILL-0005: Lazy Loading & Activation

To prevent context bloat, the system uses a two-stage activation process.

### 1. Discovery Phase (Initial Prompt)
The `PromptBuilder` will list available (selected) skills in the system prompt with just their IDs and descriptions:

```
Available Skills:
- godot-ui: Specialized in building and styling Godot UI components.
- godot-validate: Tools for checking scene and script integrity.
Use 'activate_skill(name)' to access a skill's full instructions and tools.
```

### 2. Activation Phase (The Stateful Change)
A built-in tool `activate_skill(name: String)` will be provided.

**Activation is NOT just text injection.** It is a stateful transition for the current conversation session. When executed, it performs the following:

1.  **Instruction Injection (Tool Result)**: Returns the full `SKILL.md` content. This adds the expert guidance to the conversation history.
2.  **Tool Expansion (API Update)**: This is the most critical part. It registers the skill's specialized tools in the `AIRequestHandler`'s session state.
    -   In all **subsequent turns** of the conversation, the `tools` (functions) array sent to the LLM API will now include these new specialized tool definitions.
    -   Without this step, the AI might read about a tool in `SKILL.md` but would be unable to "call" it because the LLM wouldn't know the tool exists in the API schema.
3.  **Knowledge Priming (Optional)**: If the skill contains an `examples/` directory, relevant snippets or metadata from those examples can be injected into the response to give the AI a "few-shot" boost for that specific domain.

### 3. Context Persistence
Once activated, a skill remains "Loaded" for the duration of the node's current lifecycle or until the conversation is cleared. The `AIRequestHandler` maintains a list of `_active_skill_ids` to ensure tools are included in every follow-up request.

## Technical Details

### `activate_skill` Tool Implementation
```gdscript
class_name ActivateSkillTool
extends AITool

func _init():
    super("activate_skill", "Activates a specialized skill by name. Returns the skill's instructions.")

func get_parameters() -> Dictionary:
    return {
        "type": "object",
        "properties": {
            "name": { "type": "string", "description": "The name of the skill to activate." }
        },
        "required": ["name"]
    }

func execute(arguments: Dictionary) -> String:
    var name = arguments.get("name", "")
    var skill = SkillManager.get_skill(name)
    if not skill:
        return "Error: Skill '%s' not found." % name
    
    # Load and register tools dynamically
    ToolRegistry.register_skill_tools(skill)
    
    return "<activated_skill name=\"%s\">\n%s\n</activated_skill>" % [name, skill.instructions]
```

### Refactoring `AIRequestHandler`
The request handler must support dynamic registration of tools during a single conversation turn sequence.

## Skill Tool Definition and Execution

### 1. Tool Definition (The Script)
A skill tool is a standalone GDScript file (e.g., `tools/layout_helper_tool.gd`) that extends the `AITool` base class.

**Example: `layout_helper_tool.gd`**
```gdscript
extends AITool

func _init():
    # Define name and description used in the OpenAI 'tools' definition
    super("apply_layout_preset", "Applies a layout preset (e.g. 'centered', 'full_rect') to a Control node.")

func get_parameters() -> Dictionary:
    return {
        "type": "object",
        "properties": {
            "node_path": { "type": "string", "description": "Path to the Control node." },
            "preset": { "type": "string", "enum": ["centered", "full_rect", "top_wide"] }
        },
        "required": ["node_path", "preset"]
    }

func execute(arguments: Dictionary) -> String:
    var node_path = arguments.node_path
    var preset = arguments.preset
    
    # Logic to find the node and apply anchors/offsets
    var target = context_node.get_node_or_null(node_path)
    if not target or not target is Control:
        return "Error: Node not found or not a Control node."
        
    # [Implementation of preset application...]
    
    return "Successfully applied preset '%s' to '%s'." % [preset, node_path]
```

### 2. Execution Flow

1.  **AI Decides to Act**: The AI returns a `tool_calls` message requesting `apply_layout_preset`.
2.  **Registry Lookup**: The `AIRequestHandler` checks its `_active_tools` dictionary for the tool name.
3.  **Dynamic Loading**:
    - If the tool is a core tool, it's already instantiated.
    - If it's a skill tool, it was instantiated and stored in `_active_tools` when `activate_skill()` was called.
4.  **Context Injection**: The `AIRequestHandler` ensures the tool's `context_node` property is set to the current node (e.g., the `AIChat` or `AIAgentAssisted3D` instance).
5.  **Execution**: The handler calls `tool.execute(parsed_arguments)`.
6.  **Result Return**: The string returned by `execute()` is sent back to the AI in a `tool` role message.

### 3. Safety and Sandboxing
- Tools are executed within the Godot environment.
- Skills from `res://ai_skills/` (project-space) are treated with the same permissions as the user's own scripts.
- Skill tools should use the `context_node` to ensure they operate relative to the correct scene or node.

## REQ-SKILL-0006: The Skill-Creator Meta-Skill

The `skill-creator` is a specialized, built-in skill designed to bootstrap other skills.

### 1. Skill Content
- **`SKILL.md`**: Instructions on how to architect a Godot AI skill, including folder structure, `AITool` subclassing conventions, and writing effective expert instructions.
- **Tools**:
    - `create_skill_resource(name: String, description: String)`: Creates the base folder and `SKILL.md`.
    - `add_tool_to_skill(skill_name: String, tool_script_content: String, filename: String)`: Generates a new `AITool` GDScript within the skill's `tools/` folder.

### 2. Workflow
1.  User: "Create a skill for managing Godot Physics materials."
2.  AI (via `skill-creator`):
    -   Calls `create_skill_resource("godot-physics", "Tools for physics material management.")`.
    -   Analyzes the physics system and generates a tool script for tweaking friction/bounce.
    -   Calls `add_tool_to_skill("godot-physics", "extends AITool...", "physics_tweaker_tool.gd")`.
3.  The new skill is now discoverable and ready for activation.
